# Internationalization (i18n) Guide

This project uses **i18next** for internationalization on public-facing pages (persona pages and widget/embed pages). Dashboard pages are NOT translated.

## Overview

| Feature             | Implementation                                   |
| ------------------- | ------------------------------------------------ |
| Library             | `i18next` + `react-i18next`                      |
| Language Detection  | **Persona settings only** (set by persona owner) |
| Supported Languages | 14 languages (see below)                         |
| RTL Support         | Yes (Arabic)                                     |
| Works In            | Persona pages + Widget/Embed                     |
| Dashboard           | English only (not translated)                    |

## How Language Is Determined

**IMPORTANT:** Language is determined by **persona settings only**, NOT by browser detection or user preference.

```
Persona owner sets language → API returns persona.language → Widget/page uses that language
```

Flow:

1. Persona owner sets language in dashboard persona settings
2. API endpoint `/expert/{username}/public` returns `language` field
3. Widget receives language via `lang` URL parameter
4. Persona pages read language from persona data
5. Default to English (`en`) if not set or set to `"auto"`

**This means:**

- Visitors see the language the persona owner configured
- There is no language selector for end users
- Dashboard pages are always in English

## Quick Start

### Using Translations in Components

```tsx
import { useTranslation } from "react-i18next";

function MyComponent() {
  const { t } = useTranslation();

  return (
    <div>
      <h1>{t("chat.emptyState.title")}</h1>
      <p>{t("chat.emptyState.subtitle")}</p>
      <button>{t("common.continue")}</button>
    </div>
  );
}
```

### Translation Keys Structure

Keys are organized by feature/component:

```json
{
  "common": {
    "loading": "Loading...",
    "error": "Error",
    "continue": "Continue"
  },
  "chat": {
    "modeToggle": {
      "voice": "Voice",
      "text": "Text"
    },
    "connection": {
      "connecting": "Connecting...",
      "settingUp": "Setting up chat...",
      "takeMoment": "This will only take a moment"
    },
    "input": {
      "placeholder": "Type your message..."
    }
  }
}
```

Access nested keys with dots: `t("chat.modeToggle.voice")`

### Dynamic Values (Interpolation)

```json
{
  "session": {
    "timer": {
      "remaining": "{{time}} remaining"
    }
  },
  "voice": {
    "limitExceeded": {
      "conversationEnded": "Your conversation with {{name}} has ended."
    }
  }
}
```

```tsx
t("session.timer.remaining", { time: "5:00" });
// → "5:00 remaining"

t("voice.limitExceeded.conversationEnded", { name: "John" });
// → "Your conversation with John has ended."
```

## Adding a New Language

1. Create the translation file:

```bash
cp src/i18n/locales/en.json src/i18n/locales/XX.json
```

2. Translate all strings in the new file

3. Register the language in `src/i18n/index.ts`:

```typescript
import xx from "./locales/xx.json";

export const supportedLanguages = [
  "en",
  "es",
  "fr",
  "ar",
  "de",
  "it",
  "pt",
  "nl",
  "pl",
  "hi",
  "ja",
  "ko",
  "el",
  "cs",
  "xx",
] as const;

export const languageNames: Record<SupportedLanguage, string> = {
  // ... existing languages
  xx: "Language Name", // Add this
};

const resources = {
  // ... existing resources
  xx: { translation: xx }, // Add this
};
```

4. Add the language option to the backend persona settings

## Adding New Translation Keys (CRITICAL)

When adding ANY hardcoded strings to public-facing pages, you **MUST**:

1. **Never hardcode user-visible strings** - Always use translation keys
2. **Add the translation key to ALL 14 locale files** - Not just English
3. **Use the `useTranslation` hook** from react-i18next

### Step-by-Step

1. Add the key to `en.json` first with English text:

```json
{
  "myFeature": {
    "title": "My Feature",
    "description": "This is my feature"
  }
}
```

2. Add the key to ALL other 13 locale files with proper translations:

```json
// es.json
{
  "myFeature": {
    "title": "Mi Función",
    "description": "Esta es mi función"
  }
}

// fr.json
{
  "myFeature": {
    "title": "Ma Fonctionnalité",
    "description": "C'est ma fonctionnalité"
  }
}
// ... and so on for all 14 locales
```

3. Use in your component:

```tsx
const { t } = useTranslation();
<h1>{t("myFeature.title")}</h1>;
```

## Widget Integration

The widget receives language via URL parameter:

```html
<script>
  ConvoxAI.init({
    expertUsername: "john",
    personaName: "default",
    // Language is automatically fetched from persona settings
  });
</script>
```

The SDK loader:

1. Fetches persona data from `/expert/{username}/public`
2. Extracts the `language` field
3. Passes it to the iframe as `?lang=XX` parameter
4. Widget initializes i18next with that language

## RTL (Right-to-Left) Support

Arabic (`ar`) is configured as an RTL language. The `I18nProvider` automatically:

1. Sets `dir="rtl"` on `<html>` element
2. Sets `lang="ar"` attribute

For RTL-aware styling, use Tailwind's RTL utilities:

```tsx
<div className="ml-4 rtl:mr-4 rtl:ml-0">Content with RTL-aware margin</div>
```

## Best Practices

### DO

```tsx
// Use translation keys
<button>{t("common.submit")}</button>

// Use interpolation for dynamic values
<p>{t("session.timer.remaining", { time })}</p>

// Keep keys organized by feature
t("chat.input.placeholder")
t("voice.status.connected")

// Use aria-labels with translations for accessibility
<button aria-label={t("widget.closeChat")}>×</button>
```

### DON'T

```tsx
// Don't hardcode strings on public-facing pages
<button>Submit</button>;

// Don't concatenate translated strings
t("hello") + " " + t("world"); // Word order varies by language

// Don't translate user-generated content
// Persona names, chat messages, etc. stay in original language

// Don't use browser detection for language
// Language comes from persona settings only
```

## Troubleshooting

### Translations Not Loading

1. Check that `I18nProvider` wraps your component
2. Verify the translation key exists in all locale files
3. Check browser console for i18next errors
4. Ensure you're importing from the correct path (`react-i18next` or `../../i18n`)

### RTL Not Working

1. Ensure the language is in `rtlLanguages` array in `src/i18n/index.ts`
2. Check that `I18nProvider` is mounted
3. Verify `document.documentElement.dir` is set to `rtl`

### Widget Not Translating

1. Verify persona has language set in dashboard
2. Check that `I18nProvider` wraps `EmbedApp`
3. Ensure translation files are included in the Vite build
4. Check the `lang` URL parameter is being passed to iframe

### Language Showing as English

1. Check that persona owner has set language in persona settings
2. Verify API is returning the `language` field
3. Default is English when language is not set or is `"auto"`
