# Prompt Lifecycle Documentation

This document explains how prompts are created, stored, and used throughout the expert clone system.

---

## Overview

The system uses a **3-phase lifecycle** for persona prompts:

1. **Ingestion** - Collect expert knowledge
2. **Generation** - Build personalized prompts from knowledge
3. **Runtime** - Load and use prompts in conversations

---

## Phase 1: Knowledge Ingestion

### Data Sources

Persona knowledge comes from multiple sources:

- **LinkedIn**: Profile + posts
- **Twitter**: Profile + tweets
- **Website**: Homepage + pages + blog posts
- **PDF**: Documents

### API Endpoints

```
# Step 1: Create persona
POST /api/v1/ingestion/create-expert-profile

# Step 2: Add knowledge sources
POST /api/v1/ingestion/process-linkedin-data
POST /api/v1/ingestion/process-twitter-data
POST /api/v1/ingestion/process-website-data
POST /api/v1/ingestion/process-pdf-data

# Check status
GET /api/v1/ingestion/expert-status/{username}
```

### What Happens

1. Raw data is processed into **content chunks**
2. Chunks are stored in **vector database** (via LlamaRAGSystem)
3. Metadata preserved (URLs, timestamps, source type)

**Result**: Knowledge base ready for prompt generation

---

## Phase 2: Prompt Generation

### Trigger

After ingestion is complete, generate a prompt:

```bash
POST /api/v1/prompt/create-prompt-for-persona
{
  "username": "expert123",
  "template": "basic",      # or "advance"
  "expertise": "general",
  "platform": "openai"
}
```

### Generation Process

```
1. Check if prompt already exists (idempotent)
   ↓
2. Fetch persona from database
   ↓
3. Fetch ALL content chunks for this persona
   ↓
4. Fetch matching prompt template
   ↓
5. Build LLM instruction combining:
   - Persona Profile (name, role, description)
   - Template Example (format guide)
   - Raw Knowledge (first 12,000 chars from chunks)*
   ↓
6. Call OpenAI LLM with retry + quality checks
   ↓
7. Store generated prompt in database
```

**\*Content Limit Details:**
- **Hard limit**: First 12,000 characters only
- **Why 12,000?**: Balances context quality vs LLM token limits (~3,000 tokens)
- **If content exceeds**: Remaining content is truncated (not used for prompt generation)
- **Full content still used**: All chunks remain in vector DB for RAG retrieval at runtime
- **Recommendation**: Most important content (profile, recent posts) is typically first

### LLM Instruction Format

The system sends this to OpenAI:

```
You are to craft a high-quality persona prompt using:

Persona Profile:
Name: John Smith
Role: Senior Software Engineer
Description: Expert in distributed systems...

Reference Example (Template):
[Format guide from database]

Raw Knowledge Text (first 12,000 chars):
[Aggregated content from LinkedIn, Twitter, website, PDFs]

Produce the final adapted prompt in the same format.
```

**Content Sampling Strategy:**
- System concatenates ALL chunks with spaces: `chunk1 + " " + chunk2 + ...`
- Takes first 12,000 characters of concatenated text
- Chunk order matters: LinkedIn profile → Posts → Website → PDFs (most → least important)

### Storage

Generated prompt is stored in `persona_prompts` table:

- `example_prompt` - The generated prompt (main artifact)
- `introduction` - Auto-filled from persona description
- `is_dynamic` - false (for generated prompts)
- `is_active` - true

**Result**: Ready-to-use prompt stored in database

---

## Phase 3: Runtime Usage

### How Prompts Are Loaded

**Voice Agent (LiveKit)**:

1. **Initialization** (one-time per session):
   - Load prompt from database
   - Cache it in memory

2. **Per Message**:
   - Inject cached system prompt
   - Retrieve fresh RAG context for user query
   - Generate response

**Chat API** (text):
- Builds prompt fresh per request
- Similar RAG context retrieval

### Two Loading Strategies

**Static Prompt** (`is_dynamic=false`):
- Uses the `example_prompt` field as-is
- Default behavior

**Dynamic Prompt** (`is_dynamic=true`):
- Built from structured fields at runtime
- Uses: `introduction`, `thinking_style`, `area_of_expertise`, `chat_objective`, etc.

---

## Two Prompt Types

### Static Prompts (Default)

**How it works**:
- Generated by OpenAI LLM during creation
- Stored in `example_prompt` field
- Used as-is at runtime

**Pros**:
- Clean, AI-optimized
- No manual intervention
- Quick setup

**Use when**:
- Default expert setup
- Want automated prompt generation

---

### Dynamic Prompts (Advanced)

**How it works**:
- Built at runtime from structured database fields
- Composed from: `introduction`, `thinking_style`, `area_of_expertise`, `chat_objective`, `objective_response`, `example_responses`

**Pros**:
- Full manual control
- Discovery-based questioning flow
- Voice-specific optimizations

**Use when**:
- Need structured conversation patterns
- Want to control prompt sections individually
- Building consultant-style personas

---

## Prompt Updates

### Regenerate from Latest Knowledge

```bash
PUT /api/v1/prompt/update-prompt-for-persona
{
  "username": "expert123",
  "template": "advance"
}
```

**When to use**:
- After adding new knowledge (LinkedIn, Twitter, PDFs)
- Want to incorporate latest content into prompt
- Switching templates (basic → advance)

---

### Update Individual Fields

```bash
PATCH /api/v1/prompt/update-prompt-parameter
{
  "username": "expert123",
  "field": "chat_objective",
  "value": "Help users debug distributed systems issues"
}
```

**Updatable fields**:
- `introduction` - Persona bio
- `thinking_style` - Communication approach
- `area_of_expertise` - Domain knowledge
- `chat_objective` - Conversation goal
- `objective_response` - Response strategy
- `example_responses` - Sample interactions
- `example_prompt` - The generated prompt itself
- `is_dynamic` - Toggle static/dynamic mode
- `is_active` - Enable/disable prompt

**When to use**:
- Fine-tuning without full regeneration
- Tweaking conversation style
- Updating objectives

## Performance Optimization

### System Prompt Caching

**Voice Agent**:
- System prompt cached **once** during session initialization
- Reused for all messages in that session
- Avoids rebuilding on every user message

**Why it matters**:
- System prompt is static per session
- Improves response latency
- Reduces unnecessary computation

### RAG Context (Not Cached)

- Retrieved **fresh** for each user query
- Top 3-5 relevant chunks from knowledge base
- Injected as temporary context
- Ensures up-to-date, relevant information

## Complete Flow Example

```
1. INGESTION PHASE
   Day 1: Create persona (POST /create-expert-profile)
   Day 1: Process LinkedIn data → 50 chunks
   Day 2: Process Twitter data → 200 chunks
   Day 3: Process website data → 15 chunks

   ✅ Total: 265 chunks in vector database

2. GENERATION PHASE
   Day 3: Generate prompt
   - Fetch all 265 chunks
   - Aggregate first 12k characters
   - Send to OpenAI with template
   - Store generated prompt

   ✅ Prompt ready in database

3. RUNTIME PHASE (Voice Call)
   Session starts:
   - Load prompt from DB
   - Cache in memory

   User: "Tell me about distributed systems"
   - Use cached system prompt
   - Retrieve 3 relevant chunks
   - Generate response

   User: "How do you handle failures?"
   - Same cached system prompt
   - Retrieve 3 NEW relevant chunks
   - Generate response

   ✅ Consistent persona with dynamic knowledge

4. UPDATE PHASE (Week later)
   New LinkedIn posts added → +20 chunks
   Regenerate prompt:
   - Now uses 285 total chunks
   - Updated prompt includes new knowledge

   ✅ Prompt stays current
```

## Best Practices

### When to Regenerate Prompts

✅ **Do regenerate** when:
- Adding significant new knowledge (50+ chunks)
- Changing expert's role or focus area
- Switching from basic to advance template
- Quality issues with current prompt
- Adding critical profile information (LinkedIn, bio updates)

❌ **Don't regenerate** when:
- Minor updates (few tweets/posts)
- Just tweaking objectives or style (use PATCH instead)
- Testing voice settings
- Adding content that won't fit in 12k char limit (old tweets beyond recent 200)

### Content Ingestion Strategy

**Order matters for prompt generation:**

```
Priority 1 (Always in first 12k):
  ├─ LinkedIn profile (name, role, about, experience)
  ├─ Persona description
  └─ Recent LinkedIn posts (last 10-20)

Priority 2 (Likely in first 12k):
  ├─ Website homepage and about page
  ├─ Recent tweets (last 50-100)
  └─ Key blog posts

Priority 3 (May exceed 12k):
  ├─ Old social media posts
  ├─ Additional website pages
  └─ Large PDF documents
```

**Tip**: Ingest high-value content first, bulk historical data later

### Choosing Template Type

**Basic Template**:
- General conversational style
- Works for most experts
- Less structured, more natural

**Advance Template**:
- Discovery-based questioning
- Consultant-style interaction
- Detailed behavioral rules
- Better for professional coaching personas

### Prompt Quality Checks

**Implementation**: `app/api/prompt_routes.py` → `call_openai_llm()` function (lines 220-306)

**Quality Filters** (hardcoded, not configurable):

1. **Length Check**:
   - Minimum: 40 characters
   - Rejects: Empty or too-short responses

2. **Hallucination Markers**:
   - Detects: "As an AI language model", "I cannot access", "I don't have access to real-time", "Assistant:"
   - Action: Rejects and retries

3. **Repetition Check**:
   - Maximum: 6 occurrences of persona name
   - Example: "As John" repeated 7+ times → rejected

4. **Metadata Detection**:
   - Rejects responses containing meta-commentary
   - Example: "Here is the prompt..." → rejected

**Retry Behavior**:
```
Attempt 1: Generate → Quality check → Pass ✅ OR Fail ❌
  ↓ (if failed)
Wait 2 seconds
  ↓
Attempt 2: Generate → Quality check → Pass ✅ OR Fail ❌
  ↓ (if failed)
Wait 4 seconds (2 × attempt number)
  ↓
Attempt 3: Generate → Quality check → Pass ✅ OR Fail ❌
  ↓ (if all 3 failed)
Returns None → HTTP 500 "Failed to generate prompt"
```

**After 3 Failed Retries**:
- API returns: `500 Internal Server Error`
- Message: `"Failed to generate prompt"`
- No partial result saved
- User must fix underlying issue and retry request

**Common Causes of Retry Failures**:
- Template example is too generic or unclear
- Persona description conflicts with knowledge chunks
- Knowledge chunks contain too much noise/irrelevant data
- OpenAI API issues (rate limits, timeouts)

**Configuration**:
- ⚠️ Quality thresholds are **hardcoded** (cannot be changed via API)
- To modify: Edit `call_openai_llm()` function in `prompt_routes.py`
- Max retries: Hardcoded to 3
- Backoff timing: `2 seconds × attempt_number`

## Troubleshooting

### "Prompt already exists" on creation

**Issue**: Endpoint is idempotent - won't regenerate if prompt exists

**Solution**: Use `PUT /update-prompt-for-persona` to force regeneration

---

### Prompt doesn't include latest knowledge

**Check**:
1. Were chunks actually ingested?
   - Check: `GET /api/v1/ingestion/expert-status/{username}`
   - Look for `total_chunks_processed` count
2. Is prompt cached from before update?
   - Restart agent or reload session
3. Need to regenerate?
   - Run: `PUT /api/v1/prompt/update-prompt-for-persona`

---

### Voice agent not using my dynamic fields

**Check**: `is_dynamic` flag in database

- `is_dynamic=false` → Uses `example_prompt` (static)
- `is_dynamic=true` → Builds from structured fields (dynamic)

**Solution**: Update flag via:
```bash
PATCH /api/v1/prompt/update-prompt-parameter
{
  "username": "expert123",
  "field": "is_dynamic",
  "value": true
}
```

---

### Prompt quality is poor

**Common causes**:
- Not enough ingested knowledge (< 10 chunks)
- Template doesn't match expertise area
- Persona description is too generic
- Important content not in first 12,000 chars

**Solutions**:
- Add more knowledge sources (LinkedIn, Twitter, website)
- Try different template (basic vs advance)
- Improve persona description with specific details
- Ensure key content is ingested first (LinkedIn profile before old tweets)
- Regenerate after fixes

**Note on content limit**:
If you have 100+ chunks (50,000+ chars), only the first 12,000 chars are used for prompt generation. The prompt won't reflect content beyond this limit. However, ALL content remains in the vector database and will be retrieved during conversations via RAG.

---

### "Failed to generate prompt" error (HTTP 500)

**Issue**: OpenAI LLM failed quality checks 3 times in a row

**Debug steps**:
1. Check the logs for quality check failures:
   - Look for: "LLM output low quality on attempt X"
   - Shows which filter failed (length, hallucination, repetition)

2. Verify template quality:
   - Ensure template example is clear and not too generic
   - Check: `SELECT template, example FROM prompt_templates WHERE type='basic' AND is_active=true`

3. Check persona data quality:
   - Too much noise in chunks?
   - Conflicting information between persona description and chunks?

4. OpenAI API issues:
   - Rate limits hit?
   - Model availability issues?
   - Check OpenAI status: https://status.openai.com/

**Workarounds**:
- Wait a few minutes and retry (may be temporary API issue)
- Simplify persona description
- Use different template type
- Reduce content chunks to most relevant only
- Contact support if persistent

**Code reference**: Quality check logic in `app/api/prompt_routes.py:249-265`

## Summary

**The lifecycle in one sentence**:
Ingest expert knowledge → Generate personalized prompt from that knowledge → Cache and use prompt at runtime with dynamic RAG context.

**Key insight**:
System prompt is **static and cached** (persona personality), while RAG context is **dynamic per query** (relevant knowledge).
